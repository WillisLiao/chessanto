## What Chessanto is

A native macOS SwiftUI chess analysis and training app: import games (PGN or chess.com), analyze with a local UCI engine, get a rule-based + optional local-LLM ("Coach") coaching report per game, and practice the game's own key mistakes via spaced repetition.
Local-first: SQLite via GRDB, a local UCI engine, and an optional local Ollama model for coaching prose - no cloud backend.

Project layout: `App/` (the SwiftUI app target), `Packages/` (Swift packages: `ChessCore`, `AnalysisKit`, `CoachKit`, `EngineKit`, `ChessComKit`, `Persistence`), `handoffs/` (living state + forward-looking execution plans), `devlogs/` (dated append-only session logs).
`Chessanto.xcodeproj` is generated by XcodeGen from `project.yml` and is gitignored - run `xcodegen generate` after adding or removing any source file, before building.

## What phase 3's three items actually are (from the stub - verify before trusting)

**Item A - sidebar select, delete, pin, favorite.** Delete already works end-to-end in the data layer (`GameLibrary.delete(_:)` → `GameStore.deleteGame(id:)`, cascading via foreign keys) and is simply not wired to any UI. Pin/favorite have no data model at all as of the last migration checked (`v5_trainingIndexes` in `Packages/Persistence/Sources/Persistence/Schema.swift`) - adding them means a new forward-only `v6` migration. **Never edit a shipped migration file; only add new ones.** The sidebar (`ContentView`'s `List(library.games, selection: $selectedGameID)`) is currently single-select (`Int64?`); multi-select needs that to become a `Set<Int64>` and the detail pane's derivation reconciled with it. Cascading delete across five tables is irreversible - the stub explicitly asks you to decide whether to require typed/counted confirmation, offer undo, or both, and to make that decision explicit rather than defaulting to whatever is easiest to build.

**Item B - a genuinely insightful player dashboard.** This is confirmed (the stub verified it live) to be an enhancement of an existing feature, not new construction - `computeDashboard` already exists and already produces `AccuracyPoint`, `ThemeCount`, `MoveClassificationCount`, recomputed live with no rollup table (measured at ~3.55ms/game in release mode, so more computation on the same pipeline is affordable). The real gap: `ThemeCount` only ever surfaces three themes (`punishment`, `missedMate`, `allowedMate`) because that's all `TrainingCardFactory.themes(for:)` currently emits - a thin basis for "where you're weak." The stub lists candidate additions (mistake clustering by game phase, accuracy split by color, accuracy by time control, opening-specific results via the existing `OpeningBook`, tactical-vs-positional eval-swing source) that must each be checked against what `GameReport`/`ReportInput` actually carry before being promised in a design. Also decide whether the Progress entry point itself (currently an unlabeled chart glyph in the sidebar) needs to become more discoverable.

**Item C - chess.com identity confirmation.** The stub already called the real `https://api.chess.com/pub/player/<username>` and `.../stats` endpoints live and confirmed usernames are unique (no disambiguation picker needed - just a confirmation step showing the matched account before committing). `ChessComProfile` in `Packages/ChessComKit` currently decodes only `username`/`name`/`country`; adding `avatar`/`url` is additive, no migration (only the username string is persisted, not the fetched profile). `ChessComUsernameField` already does a real existence check and shows text confirmation - the gap is showing the *account itself* (avatar, real name, current rating) rather than just a success string. Two open questions the stub explicitly defers to you: whether this lives in onboarding only or is also re-editable from Settings (there's an existing shared `CoachSetupView` pattern to follow either way), and whether fetching/showing an avatar image at all is acceptable given the app is otherwise fully local with no other routine outbound image request - the stub says this is privacy-relevant and should be surfaced to the user as an explicit question, not silently decided.

## Design system and conventions you must match

- Dark, brass-accent, minimal analysis-board aesthetic. Reuse the existing tokens: `DesignColors`, `DesignSpacing`, `DesignShape` (in `App/Sources/Chessanto/DesignSystem/`), and the existing `Card`/`SectionHeader`/`ClassificationChip` view idioms. Do not invent a new visual language for something an existing token already expresses correctly.

- Every accessibility label, hit target, and keyboard path added in phase 1 was verified by actually clicking/tabbing through the built app, not just by reading the SwiftUI source - macOS-native apps get this wrong in ways that only show up at runtime (see the phase 1 section of `handoffs/HANDOFF.md` for a concrete example: a `Button` that visually looked like it wrapped a whole content block but structurally only wrapped one line of it).
- Native, system-appropriate controls: this app uses standard SwiftUI `Button`/`Picker`/`List` controls styled via the design tokens, not custom-drawn chrome, and follows macOS conventions (keyboard shortcuts like `.cancelAction` on cancel/exit buttons, `.contextMenu` for secondary actions, `.accessibilityLabel` on every icon-only control).

## Database safety - mandatory before any live/native testing

Never run native QA against the live database at `~/Library/Containers/com.chessanto.app/Data/Library/Application Support/Chessanto/chessanto.sqlite`.
Copy it out (read-only use) or copy it into a QA path inside the sandboxed container and launch the app pointed at the copy:

```
LIVE="$HOME/Library/Containers/com.chessanto.app/Data/Library/Application Support/Chessanto/chessanto.sqlite"
STAMP=$(date +%Y%m%d-%H%M%S)
cp "$LIVE" "$LIVE.before-uiux-phase3-$STAMP"
md5 -q "$LIVE" > /tmp/chessanto-live-md5-before.txt
QA="$HOME/Library/Containers/com.chessanto.app/Data/tmp/uiux-phase3-$STAMP"
mkdir -p "$QA"
cp "$LIVE" "$QA/chessanto.sqlite"
open -n "<Release build path>/Chessanto.app" \
  --env CHESSANTO_ENABLE_QA_DATABASE_OVERRIDE=1 \
  --env CHESSANTO_DATABASE_PATH="$QA/chessanto.sqlite"
```

At the end of your session, `md5 -q "$LIVE"` must equal the value you recorded before starting.
If it does not, stop and figure out why before doing anything else - do not proceed as if it's fine.
