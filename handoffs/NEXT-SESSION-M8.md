## Known gaps carried forward from M7 (not blocking M7's own acceptance, but relevant polish/robustness targets for M8)

- **`OllamaClient` has no request timeout.** A genuinely-unresponsive-but-not-torn-down Ollama process (verified live via `kill -STOP`, not a hypothetical) hangs a chat turn indefinitely with no client-side recovery - `kill -9` recovers fast in this dev environment only because Ollama.app auto-respawns almost instantly (both a `launchctl` LaunchAgent and its own Electron menu-bar process), which isn't something a shipped app should rely on. Worth a real fix: add a reasonable `URLSession` timeout (or per-request timeout override) to `OllamaClient`'s chat/evaluate calls, with `CoachChat`/`CoachNarrator`'s existing fallback ladder already correctly wired to catch a thrown timeout error - the fix is narrow (client-side), not architectural.
- **The M5 AX-label gap is still only partially understood.** M7 tried `List` for the chat message view specifically to test whether it fixes the gap `MoveListView`/`GameReportView`'s bare `ScrollView`/`VStack` buttons have (button/text content not exposed via AX `name`). Result: `List` does produce a real `AXOutline`/`AXRow`/`AXCell` structure (row selection/count is queryable), but row `Text` content and per-row buttons still report no `name` via `osascript`/System Events' `name of` query. Try `AXValue` or other attributes instead of `name` before concluding this is unfixable - it may just be the wrong AX property being queried, not a real SwiftUI/AppKit limitation.
- Prose-quality-not-safety with non-frontier local models (`qwen3:0.6b`, spot-checked `qwen2.5-coder:7b`) is expected and documented (M6's own residual-risk note); not something to "fix" via prompt engineering unless the user asks specifically for coaching-quality work, which PLAN.md scopes as out of M8.
- `-0.0` eval label quirk, "Starting engine..." toolbar state never caught live in automation, and M3/M4 UI leftovers - all still open from earlier milestones, listed for completeness, not specifically M8's job unless picked up along the way per this project's "be picky about UI, fix things you notice" convention (`~/.claude/CLAUDE.md`).

## Working style notes (carried forward; they keep paying off)

- Do an M6/M7-style prep pass before writing code if this session has the budget for it - reading every touched file in full and verifying claims live (not guessing) is what made M6/M7's execution sessions go smoothly with real bugs caught early instead of late.
- Real E2E through the built app via `osascript`/System Events **AX-element** references only; raw pixel clicks and screenshots are blocked in this sandbox. - After adding/removing files under `App/`, rerun `xcodegen generate`.
- Debug on stderr, never stdout (the engine hijacks stdout); Release builds for anything timing-sensitive.
- The sandboxed DB for E2E checks: `~/Library/Containers/com.chessanto.app/Data/Library/Application Support/Chessanto/chessanto.sqlite`.
