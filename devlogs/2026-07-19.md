### Step 1: engine-side generation isolation

Used the `diagnosing-bugs` skill to independently reproduce F1 and F2 rather than trust the plan's own reproduction, per its explicit instruction.

F1 (`searchOneShot` arms its waiter after sending `go`) was confirmed by direct code reading: `EngineService.swift` sent `go` before installing the waiting continuation, and `routeBatch`'s optional chaining on a nil continuation silently no-opped when the terminating bestmove arrived first.
This is a structural ordering bug, not something that needed a timing race to confirm.

F2 (cross-position score contamination) was reproduced live.
Extended `Packages/EngineKit/Sources/engine-smoke/main.swift` with a repeated assertion: run a midgame search, then immediately a bounded search on a stalemate position with no legal moves, and assert no scored info leaks into the stalemate search.
This failed on the first attempt, 100% of the time in this session's testing (the plan's own diagnosis reported 2 of 3 runs), with a scored info line carrying a real midgame PV (e.g. `cp=27 pv=d7d6 b1c3 f8e7`) attributed to the stalemate search's generation.

Implemented the plan's specified fix: added `searchGeneration` to `AnalysisEngine`, set on every `go` call and reset to `0` in `markSearchEnded`, with the listener task stamping updates with `searchGeneration` instead of `generation`.

This fix alone did not close the window.
Repeated testing after the fix still reproduced contamination, now traced to a deeper root cause than the plan diagnosed: chesskit-engine's `Engine.swift` (`setMessengerResponseHandler`) dispatches each raw engine response through its own unstructured `Task`, with no ordering guarantee across responses.
A stray response from the just-finished search can still be processed by our listener task after `searchGeneration` has already been bumped for the new search, because the actor-hop for that stale response's processing can be scheduled after the new search's `go` call even though the underlying response logically arrived first.

Verified this independently with an isolated single-search harness against the stalemate FEN: Stockfish itself legitimately emits a final unscored (or `cp=0`, empty-PV) summary line before `bestmove (none)` for a genuinely terminal position, which is not contamination.
The initial version of the extended `engine-smoke` assertion was too strict and flagged this legitimate output; corrected it to only flag a scored info with a non-empty PV or a mate value, since a position with zero legal moves cannot legitimately produce either.

Added a mitigation beyond the plan's specified fix: `AnalysisEngine.setPosition` now sends UCI `isready` and waits for `readyok` (a new `resumeReady`/`waitUntilReady` pair, using chesskit-engine's existing `EngineResponse.readyok` case) before bumping the generation, followed by a 30ms settle delay.
This is documented in code as an empirically-verified mitigation, not a mathematical proof, because `readyok`'s own delivery goes through the same unstructured-Task dispatch as everything else.
After this change, 14 consecutive `engine-smoke` runs (70 repeated iterations of the generation-isolation assertion) passed clean.

Also corrected the `setPosition` doc comment for F5 (the real stop-wait bound is ~300ms via a 10ms-poll loop capped at 300 iterations, not "roughly three seconds" as the phase-1 backlog document stated).

Gate passed: `swift test --package-path Packages/EngineKit` and `swift run --package-path Packages/EngineKit engine-smoke`, both green, engine-smoke run repeatedly with no failures.

### Step 3: rewiring `EngineService`

Deleted `batchCollector`, `batchGeneration`, `batchContinuation`, and `awaitBatchSearch`.
Installed a single `activeSearch: BoundedSearchSession?`.
`searchOneShot` now creates and installs the session before sending `go`, races `session.value()` against a deadline task (`movetimeMilliseconds * 4 + 3000`, matching the plan's formula exactly for all four callers: 3400ms/4400ms/11000ms/5000ms), and sends `engine.stop()` plus clears `activeSearch` only on the failure paths (timeout or cancellation), not on normal completion.

Fixed F4 (`coachEvaluate`'s cancellation not reaching the queued work) with a new `runOnFIFOTail` helper shared by `coachEvaluate` and the new `evaluateTrainingPosition`.
It wraps the FIFO-chained work task in `withTaskCancellationHandler`, calling `workTask.cancel()` on the caller's cancellation, which propagates cancellation into `searchOneShot`'s own cancellation handler because the work task calls directly into it rather than spawning further unstructured work.
`coachEvaluateTail` is always reassigned to a task that awaits the work task's outcome regardless of success, failure, or cancellation, so the FIFO can never wedge behind an abandoned call now that `searchOneShot` itself is bounded.

Added `evaluateTrainingPosition` (still returning the old `TrainingEngineEvaluation` type at this step, formalized to `WhitePerspectiveScore` in Step 4) that shares `coachEvaluateTail` but does not call `coachEvaluate`, per the plan's requirement that the training domain not couple to the Coach tool's `EngineToolResult` shape.

### Step 5: grading correctness (F6 and F7)

Implemented the terminal-position short circuit specified in the plan: a checkmating attempted move (via `ChessGame.replayLine`'s `isCheckmate`) short-circuits to `.mate(1)`/`.mate(-1)` oriented by mover with no engine call, and a stalemating move (no legal moves, not in check) short-circuits to `.centipawns(0)`.
Stalemate detection needed a small FEN piece-placement parser (`occupiedSquares(ofColor:fen:)`) since `ChessGame` exposes `legalMoves(from:at:)` but not direct board/position access from the App target.

Replaced F6's blanket cached-line acceptance: a move matching any cached ranked line's first move is now graded against that line's own cached score via the normal `classify` path, not force-accepted as `.strong`.
Rank one matching itself still yields a loss of 0 and grades `.strong`, verified by `cachedRankOneRemainsStrongWithoutEngineSearch`.

Replaced F7's mate handling with the plan's four-bullet total comparison over `WhitePerspectiveScore`, oriented once via `oriented(forMover:)`.
The "best had mate, mover has centipawns" bullet needed one judgement call the plan left unspecified: what counts as "clearly winning" enough to grade `.inaccurate` rather than `.incorrect`.
Chose 200 centipawns (documented inline) as a defensible round threshold; the plan did not prescribe an exact number and none of the ten new tests depend on the precise value beyond confirming the branch is reachable and not `.strong`.

Added all 10 named tests to `App/Tests/ChessantoTests/TrainingDomainTests.swift`, including two tests using a real, verified back-rank checkmate (`6k1/5ppp/8/8/8/8/8/R6K w - - 0 1`, `a1a8`) and a real, verified stalemate (`7k/8/6K1/8/8/8/5Q2/8 w - - 0 1`, `f2f7`, producing the exact same resulting position independently confirmed live against Stockfish during Step 1's diagnosis).

### Step 6: recoverable grading failure

Added `@Published private(set) var promptError: String?` to `PracticeSessionViewModel`.
`submit` now catches `EngineSearchError` specifically (leaving other, genuinely unexpected errors routed to `.failed` as before), decrements the just-incremented `attemptsOnCurrentCard` back (a timeout is not a real graded attempt), sets a short retryable message, and returns to `.prompt` with the card and board intact.
No attempt is persisted and no scheduler advance happens on this path, since the failure occurs before either of those calls.

`PracticeSessionView`'s prompt controls render `promptError` in `DesignColors.error` above the Hint/Reveal buttons when present.

Added all 3 named tests to `PracticeSessionViewModelTests.swift`; all passed on the first implementation, including a retry test that flips a captured `shouldTimeout` flag between two `submit` calls on the same card and confirms the second grades normally with exactly one persisted attempt.

### Corrections found

- The stub's "the actual bad continuation that was played" cannot be sourced from `PunishmentFact.refutingSAN`, despite being the obvious adjacent candidate: that fact's own doc comment says it fires from "the rank-1 PV of the post-move position," i.e. the engine's recommended reply, not a record of what the opponent actually played. The real source is the game's own mainline, replayed from the moment's ply via `GameReplayViewModel`'s already-loaded `chessGame`.
- `GameReplayViewModel.adoptLine(sanMoves:)` unconditionally persists every played move to the `variation` table (`recordVariationMove` → `store.insertVariationMove`). A preview must not reuse this path or a parallel copy of it; the plan's `LinePreviewController` is deliberately built on the DB-free `ChessGame.replayLine(fromUCI:startingFEN:)` primitive instead, with zero dependency on `GameStore` or `GameReplayViewModel`.
- `CoachVerifier.numericClaimViolations` scans the whole response text for eval/mate/percent-shaped substrings and checks them against known values; it does not require the parenthetical-after-a-cited-line format `CoachPrompt.groundingRules` asks the model for. This means a rendering-only fix (splitting already-verified narration text into separate sentence lines) cannot weaken grounding, since it never touches the raw string the verifier already checked.

### Design decisions locked

Used the `codebase-design` skill to settle the seam: `LinePreviewController` (`App/Sources/Chessanto/Board/`), a small ObservableObject wrapping a precomputed `[ReplayedMove]` plus a step index, owned independently by `PracticeSessionViewModel` and by `GameReplayView` (never by `GameReplayViewModel`), constructed from two different real data sources - `TrainingCard.rankedLines` for the better line, and a new small pure `GameReplayViewModel.uciContinuation(fromPly:maxPlies:)` for the actual continuation.
Used `ui-ux-pro-max` for the presentation decisions: step-through only (no autoplay), reusing the existing `theme.highlight` board token for the previewed move (no new arrow color) and the existing step-control/bordered-button idioms (no new visual vocabulary), no preview affordance before a practice card is answered (would defeat the phase 1 hint gate), and sentence-level chunking (not truncation, not a new disclosure affordance) to resolve the "too dense" vs "no modal/tooltip/disclosure" tension named in the original scoping stub.

Full detail, every fixed design decision, the build sequence, and the acceptance scenarios are in `handoffs/NEXT-SESSION-UIUX-CLARITY-PHASE-2.md` itself.
