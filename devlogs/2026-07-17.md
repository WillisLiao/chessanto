# 2026-07-17

## M2 prep: live engine verification and execution plan

Goal of this session: turn `handoffs/NEXT-SESSION-M2.md` from a bootstrap
brief into a self-contained execution plan a fresh session can follow, and
de-risk its biggest unknown (no live Stockfish run had ever happened in
this repo) before that session starts.

### Read everything against real source

Re-read PLAN.md, all M1 code, and the actual checked-out sources of
chesskit-engine 0.7.0 and chesskit-swift (`.build/checkouts/`), which
surfaced four load-bearing facts the M1 handoff didn't know:

- chesskit-engine compiles Stockfish with `NNUE_EMBEDDING_OFF`, so the
  M1-era belief that "Stockfish 17 has an embedded default network" is
  wrong for this build. Worse, Stockfish 17 has no classical-eval
  fallback: a search with no net loaded prints "The engine will be
  terminated now." and calls `exit(EXIT_FAILURE)` in-process - it would
  have killed the app on first analysis. This is also (with the run-loop
  point below) why upstream's own Stockfish tests are disabled since 17.
- `Engine.start()` dup2()s the process's stdout into the engine's read
  pipe: `print()` stops reaching the terminal once the engine runs.
- Engine responses arrive via NSFileHandle run-loop notifications on the
  main thread: block the main thread and responses never arrive.
- The dup2 is process-wide, so only one live engine instance can exist
  per process - batch analysis and live scrubbing must share one engine.

Also confirmed: UCI scores are side-to-move relative (not white-relative),
`EngineResponse.Info.multipv` carries the MultiPV rank (which
`AnalysisEngine.EngineInfo` was dropping), `EngineCommand.go` supports
`movetime`, and chesskit-swift ships a public `EngineLANParser` (UCI ->
`Move`, castling already king-from/to) that M2 can use for PV -> SAN.

### Fixed and built

- `scripts/fetch-nnue.sh`: downloads both Stockfish 17 nets into
  `App/Resources/` and verifies them against the sha256 prefix embedded
  in their filenames (75MB + 3.5MB, gitignored via `*.nnue`).
- `project.yml`: the target's `resources:` key was silently dead -
  XcodeGen has no such key and ignores unknown keys, so the app never had
  a resources build phase (M1 had no resources, so nothing noticed).
  Replaced with a `sources` entry using `buildPhase: resources` plus
  excludes for Info.plist/entitlements; verified both nets now land in
  `Chessanto.app/Contents/Resources/`, where chesskit-engine's
  `Bundle.main` auto-lookup expects them.
- `EngineKit`: `EngineInfo` now forwards `multiPVRank`; added
  `setOption(name:value:)`; added the `engine-smoke` executable target -
  a live smoke run designed around the landmines above (stderr-only
  output, synchronous main + `RunLoop.main.run()`, detached engine task,
  explicit `EvalFile` setoptions since a CLI has no `Bundle.main` nets).

## M2 execution: engine integration and move classification

Followed `handoffs/NEXT-SESSION-M2.md` step by step in a second session the
same day; every step's verification passed before moving to the next.

- **Persistence**: `AnalysisRecord` (mirrors the existing `analysis` table)
  plus `GameStore.saveAnalysis/analysis/analyzedPlyIndices/deleteAnalysis`,
  all async, delete-first replacement per `(gameId, plyIndex)`. 6 new tests.
- **EngineKit**: `go(movetimeMilliseconds:)`, and closed the stale-generation
  window from the prep plan (fact 8) - `setPosition` now waits for an
  in-flight search's terminating `bestmove` (`isSearching` flag flipped by
  the listen task) before bumping the generation, so nothing from the old
  search can arrive tagged with the new one. `EngineInfo` gained a public
  memberwise init (needed for structural tests outside the package).
  `engine-smoke` extended with a movetime-based search assertion; still
  exits 0 (`OK: live Stockfish verified...`).
- **ChessCore**: `ChessGame.uciMove(at:)` turned out to be a one-liner -
  chesskit-swift's `Move.lan` already produces exact UCI notation including
  castling and promotion letters, so no new logic was needed there.
  `ChessGame.sanLine(fromUCI:startingFEN:)` replays a UCI line via
  `EngineLANParser.parse` + `Board.move`/`completePromotion`. 4 new tests,
  including a crafted promotion PGN and the smoke run's own PV round-tripped
  into SAN.
- **AnalysisKit**: `WinProbability`, `MoveClassification`,
  `MoveClassifier`, `Accuracy` - pure math, no engine/DB imports, exactly
  the formulas fixed in the prep plan. 17 tests, including the worked
  example (+50cp/-250cp -> Mistake), a Black-to-move blunder (proving the
  classifier correctly flips perspective rather than reading white-relative
  cp directly), missed-win firing/not-firing, and mate-score plies. Band
  boundary tests use bisection to invert the winP formula back to cp, with
  a small margin (0.5 rather than the plan's 0.01) away from each threshold
  since integer-cp resolution can't hit exact fractional win probabilities.
- **App**: `EngineService` (`@MainActor`, one `AnalysisEngine` for the whole
  app, live infinite analysis and batch analysis mutually exclusive per
  engine facts 5/6). Batch search completion uses a
  `withTaskCancellationHandler` around a `CheckedContinuation` that the
  routing loop resumes on the terminating bestmove (including
  `bestmove "(none)"`), with `engine.stop()` on cancellation so the
  continuation still resolves before the batch loop rethrows. `BatchCollector`
  and `LiveGenerationFilter` factored out as plain structs, unit-tested
  without a live engine (9 App tests total). Wired `EvalBarView`,
  `EvalGraphView`, move-list classification badges, an Analyze/Re-analyze
  toolbar with a quality picker and cancellable progress, and a
  display-only MultiPV lines panel into `GameReplayView`/
  `GameReplayViewModel`. Live analysis starts on ply change (200ms debounce)
  and stops on view disappear; the eval bar/lines panel only trust
  `EngineService.liveEvaluation` when its `fen` matches the ply being shown,
  falling back to the cached DB row otherwise - this is what makes the
  no-stale-eval acceptance drill pass.
