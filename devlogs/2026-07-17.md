## M2 prep: live engine verification and execution plan

Goal of this session: turn `handoffs/NEXT-SESSION-M2.md` from a bootstrap
brief into a self-contained execution plan a fresh session can follow, and
de-risk its biggest unknown (no live Stockfish run had ever happened in
this repo) before that session starts.

### Read everything against real source

Re-read PLAN.md, all M1 code, and the actual checked-out sources of
chesskit-engine 0.7.0 and chesskit-swift (`.build/checkouts/`), which
surfaced four load-bearing facts the M1 handoff didn't know:

- chesskit-engine compiles Stockfish with `NNUE_EMBEDDING_OFF`, so the
  M1-era belief that "Stockfish 17 has an embedded default network" is
  wrong for this build. Worse, Stockfish 17 has no classical-eval
  fallback: a search with no net loaded prints "The engine will be
  terminated now." and calls `exit(EXIT_FAILURE)` in-process - it would
  have killed the app on first analysis. This is also (with the run-loop
  point below) why upstream's own Stockfish tests are disabled since 17.
- `Engine.start()` dup2()s the process's stdout into the engine's read
  pipe: `print()` stops reaching the terminal once the engine runs.
- Engine responses arrive via NSFileHandle run-loop notifications on the
  main thread: block the main thread and responses never arrive.
- The dup2 is process-wide, so only one live engine instance can exist
  per process - batch analysis and live scrubbing must share one engine.

Also confirmed: UCI scores are side-to-move relative (not white-relative),
`EngineResponse.Info.multipv` carries the MultiPV rank (which
`AnalysisEngine.EngineInfo` was dropping), `EngineCommand.go` supports
`movetime`, and chesskit-swift ships a public `EngineLANParser` (UCI ->
`Move`, castling already king-from/to) that M2 can use for PV -> SAN.

### Fixed and built

- `scripts/fetch-nnue.sh`: downloads both Stockfish 17 nets into
  `App/Resources/` and verifies them against the sha256 prefix embedded
  in their filenames (75MB + 3.5MB, gitignored via `*.nnue`).
- `project.yml`: the target's `resources:` key was silently dead -
  XcodeGen has no such key and ignores unknown keys, so the app never had
  a resources build phase (M1 had no resources, so nothing noticed).
  Replaced with a `sources` entry using `buildPhase: resources` plus
  excludes for Info.plist/entitlements; verified both nets now land in
  `Chessanto.app/Contents/Resources/`, where chesskit-engine's
  `Bundle.main` auto-lookup expects them.
- `EngineKit`: `EngineInfo` now forwards `multiPVRank`; added
  `setOption(name:value:)`; added the `engine-smoke` executable target -
  a live smoke run designed around the landmines above (stderr-only
  output, synchronous main + `RunLoop.main.run()`, detached engine task,
  explicit `EvalFile` setoptions since a CLI has no `Bundle.main` nets).

## M3: Exploration Mode (variation play, tree, lines-panel adoption)

Same day, third session. Followed `handoffs/NEXT-SESSION-M3.md`'s bootstrap
approach: re-derived the design from chesskit-swift's actual `MoveTree`
source rather than trusting assumptions.

### Design decision: don't fight chesskit's `MoveTree` internals

`MoveTree.Node.children` is internal to ChessKit and there is no public
API to enumerate branch alternatives or delete a subtree. Rather than
trying to mutate chesskit's tree, the app now owns the tree shape itself:

- `ChessGame` gained `allIndices`, `isMainline(_:)`, `parent(of:)` (derived
  from the existing public `history(for:)`), and `mainlineAncestor(of:)` -
  enough to answer "what's the parent of this index" and "where does this
  variation rejoin the mainline" without touching ChessKit internals.
- `Persistence` gained `VariationRecord` (one row per played variation
  move: `parentPlyIndex` for a row branching directly off the mainline,
  `parentVariationId` pointing at the preceding row otherwise - a plain
  parent-pointer forest) and `GameStore.insertVariationMove` /
  `variations(gameId:)` / `deleteVariation(id:)`. Deleting a row cascades
  to its whole subtree via the schema's existing `ON DELETE CASCADE` on
  `parentVariationId` - confirmed with a test that deletes a middle node
  of a 3-deep chain and asserts a sibling survives.
- `GameReplayViewModel` rebuilds this tree in memory on load by replaying
  each `VariationRecord` onto `chessGame` in insertion order (parents are
  always inserted before children, since you can't play a move before its
  parent exists) and keeps `variationChildren`/`variationNodes` alongside
  the existing mainline `moveIndices`. New moves are persisted the same
  way, one row per move, immediately after being played - matching M2's
  per-ply save pattern so quit/crash never loses more than the move in
  flight. A move that happens to match an existing mainline/variation
  continuation is recognized and not re-persisted as a duplicate branch
  - except see the known gap below.
- `ChessGame.playMove(from:to:at:)` now auto-completes promotion (default
  queen; `PromotionKind` enum) - needed since board taps are the first
  real move-input path in the app - and gained `isPromotion(from:to:at:)`
  for a future picker.
- Board taps: `BoardView` squares became real `Button`s (was
  `.onTapGesture`) with click-to-select/click-to-move and legal
  destination highlighting - see the accessibility note below for why
  this mattered beyond just code style.
- Move list: recursive `variationRows(startingAt:depth:)` flattens the
  tree so the principal (first-played) continuation stays at its parent's
  depth while any additional branch at the same point nests one level
  deeper - the standard PGN variation-rendering shape. Delete removes a
  row and everything explored from it (`GameReplayViewModel.removeSubtree`
  mirrors the DB cascade in memory). "Back to game" jumps to
  `mainlineAncestor(of: currentIndex)`. Promote/collapse from PLAN.md's
  bullet are **not** implemented this milestone - out of scope for the
  accept criterion, noted below as a gap.
- `LinesPanelView` is no longer display-only: clicking a line adopts it
  (`ChessGame.sanLine(fromUCI:startingFEN:)` -> `adoptLine(sanMoves:)`,
  which plays each move as a nested variation branch one deeper than the
  last).
- `currentFEN`/`currentEvalDisplay` were mainline-`moveIndices`-array
  lookups in M2 (fine, since nothing but the mainline existed yet) -
  generalized to read straight off `chessGame.fen(at: currentIndex)` so
  the eval bar and live engine tracking work identically for variation
  positions, which is what actually makes "eval bar tracks every explored
  position live" true rather than just true-by-coincidence on the
  mainline.

### Real E2E verification, and what it caught

Screen Recording still isn't available in this environment. The
2026-07-17 earlier-session `osascript`/System Events method mostly
carried over, with one important correction: **raw pixel-coordinate
clicks (`cliclick`, and System Events' `click at {x,y}`) are blocked in
this sandbox** (`-25211`, "not allowed assistive access") - only
AX-element-reference clicks (`click item N of buttons of ...`) are
permitted. That single fact drove a real product fix, not just a test
workaround: `BoardView`'s squares used `.onTapGesture`, which (unlike

undrivable and - worse - would have been invisible to VoiceOver too.

adopt-a-line buttons still don't expose as `AXButton` despite an
identical `Button` + `.buttonStyle(.plain)` shape to the (working)

a mitigation but this needs re-verification once Screen Recording (or a
proper XCUITest target) is available - the automation for this milestone
exercised the lines-panel adopt path only indirectly, not through a
confirmed click.

The introspection also caught a real layout bug before it shipped:
`MoveListView` nested each move's variation children inside
`if let san = viewModel.san(at: index)`, so variations branching directly
off the game's start position (ply 0, which has no SAN) never rendered
or were deletable. Fixed by hoisting the variation `ForEach` out of that
`if`.

## M5: rule-based coaching report

Followed `handoffs/NEXT-SESSION-M5.md` step by step (a planning session's
self-contained execution plan); every step's verification passed before
moving on. What's new:

- **ChessCore**: `ReplayedMove`, `ChessGame.replayLine(fromUCI:startingFEN:)`
  (generalizes the old `sanLine`, now also returning check/mate flags,
  captured piece, and the resulting FEN), `moveDetail(at:)`,
  `ChessGame.material(fen:)`, `ChessGame.epd(fromFEN:)`. Fixed `sanLine`'s
  stale doc comment (it does include `+`/`#`, verified in the M5 prep

- **Opening book**: `scripts/fetch-eco.sh` downloads the 5
  lichess-org/chess-openings TSVs (3,803 entries, CC0-1.0) into `eco.json`.
  `OpeningBook` replays every line through `ChessGame` and indexes final
  positions by EPD, resolving transpositions deterministically. Replaying
  the raw dataset at app-launch time measured **~7.5s** in an early
  version - well past the "well under a second" budget - so the book now
  ships a **precomputed** `eco-index.json` (a plain `{epd, eco, name}`
  array with transpositions already resolved), built once via a
  since-deleted `eco-indexer` scratch executable and loaded by
  `OpeningBook.loadFromBundle()` in ~20ms. One dataset entry ("Scandinavian
  Defense: Boehnke Gambit") can't be replayed because its final move is an
  en-passant capture and chesskit-swift's FENs omit the en-passant target
  square even when legal (the same chesskit gap M2 already flagged for
  stored analysis rows) - logged, not fixed, and pinned by name in a test
  so a *new* unreplayable entry would be caught as a real regression.
- **AnalysisKit - the Facts/detector/report pipeline** (all DB-free, only
  imports ChessCore): `ReportInput`/`PlyRecord`/`RankedLine` (the app's
  DB-free view of analysis rows), `EvalSwingFact`/`BetterMoveFact`/
  `PunishmentFact`/`MissedMateFact`/`AllowedMateFact`/`OpeningFact`,
  `ThemeDetector` (one function per fact, each replaying positions through
  `ChessGame` rather than string-matching SAN), `KeyMomentSelector`
  (blunders/missed-wins always included, filled to a cap of 8 by largest
  win-probability drop), `ReportBuilder` (assembles the whole
  `GameReport`), `ReportText` (closed templates only - no free-composed
  chess prose), `FactAuditor` (independently re-derives every fact via the
  same detectors and drops anything that fails to match - the seed of
  M6's `CoachVerifier`). `EvalLabel` extracts the eval-string formatting
  previously duplicated in `GameReplayViewModel`.
- **App**: fixed the M4-logged "Analyze button silently no-ops before the
  engine finishes starting" bug with a fourth toolbar branch (disabled
  "Starting engine..." button, gated on `!isStarted && unavailableReason
  == nil`). `GameReplayViewModel` now retains every ranked analysis row
  per ply (not just rank-1) and builds a `GameReport` whenever the game is
  fully analyzed, invalidating it on re-analyze. The right pane is now a
  segmented "Moves"/"Report" picker; `GameReportView` renders the header,

### A real bug the E2E pass caught (and fixed): stale game data on selection change

Switching the sidebar selection from one already-analyzed game straight to
another left the entire replay pane - board, move list, eval, and the new
Report tab - showing the *previous* game's data, even though the window's
`navigationTitle` correctly updated to the new game's names. Root cause:
`ContentView`'s `detail:` closure is `if let selectedGameID, let game =
...  { GameReplayView(game: game, store:) }` - since the `if` branch stays
taken across a selection change (nil -> non-nil is the only branch
transition SwiftUI's diffing sees), `GameReplayView`'s structural identity
never changes, so its `@StateObject private var viewModel` - populated
once in `init` - is never recreated for the new `game`. Fixed with
`.id(game.id)` on the `GameReplayView` call site, forcing SwiftUI to treat
each game as a distinct view identity. This is a genuine pre-M5 bug (any
milestone's session that only ever tested one game per launch would have
missed it); confirmed fixed by switching between the two analyzed fixture
games and seeing the Report tab (and the eval bar) actually update.

### A real bug the E2E pass caught (and fixed): "clean game" claimed on a game with a real blunder

The Report tab's Takeaways section said "A clean game: no mistakes or
blunders at this analysis depth" for the real 55-ply fixture game *even
though* the game had one real blunder and 3 key moments. Root cause:
`ReportBuilder.buildTakeaways` only appended the clean-game fallback when
`takeaways.isEmpty` (i.e. no recurring-punishment/mate/opening-deviation
pattern fired), not when `keyMoments.isEmpty` - a game can have real
mistakes with no *aggregate* pattern among them. Fixed by checking
`keyMoments.isEmpty` specifically, with an honest alternate fallback ("No
single recurring pattern stood out - see the key moments above for
specifics.") for the has-moments-but-no-pattern case. Regression test
added (`takeawaysDoNotClaimACleanGameWhenAKeyMomentExistsWithNoAggregatePattern`).
This is exactly the kind of false statement the M5 accept criterion
exists to catch - it would have been a genuine hallucination-shaped bug in
a rule-based system, not just an LLM.

### Live Ollama verification (v0.31.2, Ollama.app serving on 127.0.0.1:11434)

Curled every endpoint M6 will use and pasted the real transcripts into
the plan:

- Detection must probe HTTP, not the binary: this machine's Homebrew CLI
  is 0.30.10 while the serving app is 0.31.2. `GET /api/version` is the
  health check; connection refused = not running.
- `/api/tags` and `/api/show` expose a `capabilities` array
  (`completion`/`tools`/`thinking`/`vision`/`insert`) - this is how the
  model picker checks tool support before enabling the engine-tool loop.
- Chat streaming is NDJSON with per-token `message.content` deltas and a
  terminal `done:true` stats line. qwen3-family (thinking) models emit a
  separate `message.thinking` field - streamed as its own deltas, never
  `<think>` tags in content - and `"think": false` disables it (silently
  accepted by non-thinking models too, so the client sends it always).
- Tool calling: `tool_calls[].function.arguments` is a **JSON object**
  (not an OpenAI-style string); in streaming mode the complete
  `tool_calls` array arrives in a single `done:false` chunk;
  `done_reason` stays `"stop"`, so detection keys on the field's
  presence. The `role:"tool"` + `tool_name` round-trip works on 0.31.2.
- Pull-with-progress: NDJSON status lines ending in
  `verifying sha256 digest` / `writing manifest` / `success`. The first
  real pull this session ended with `{"error":"503: "}` **inside an
  HTTP 200 stream** - per-line `error` keys are the only failure signal,
  and the retry resumed the partial download. The plan's pull client
  handles both.
- The default loaded context is 4096 regardless of the model's maximum
  (observed live via `/api/ps`); `options.num_ctx` reloads correctly and
  measurably grows RAM (`size_vram` 1.0 -> 2.44 GB on a 0.6b model at
  16k). The coach always sets `num_ctx` explicitly.
- Two live hallucination specimens from qwen2.5:0.5b went straight into
  the plan's rationale: a tool call with the move list concatenated into
  the FEN argument, and a narration calling a verified "+0.35, best line

  ChessCore-replay validation before the engine ever runs; CoachVerifier
  gates every rendered sentence.

### ChessCore scratch run: the SAN-suffix trap

A real scratch package run against ChessCore verified CoachVerifier's
core primitive - SAN-line replay from arbitrary mid-game FENs via
`ChessGame(startingFEN:)` + `playMove(san:at:)` + `moveDetail(at:)` -
including rejection (nil, no crash) of illegal and nonsense tokens, and
tolerance for `Ng1f3` and `0-0` forms. It also exposed a real trap M5's
verified facts didn't cover: `playMove(san:)` does **not** derive check
state from the board - from a position where Qh5 is mate, `"Qh5#"`
reports `isCheckmate=true` but bare `"Qh5"` reports false, and spurious
suffixes (`"e4+"` on a quiet move) are accepted and echoed uncorrected.
The UCI path (`replayLine(fromUCI:)`) derives the truth (`Qh5#`,
`isCheckmate=true`) - so the plan fixes the pattern: SAN path for
legality + UCI extraction only; every board fact comes from re-replaying
the derived UCI line. The SAN<->UCI bridge round-trips exactly
(verified on a 4-move capture line).
