### Step 5 - Engine FIFO serialization

Added a `coachEvaluateTail` chokepoint to `EngineService.coachEvaluate`.
**First implementation was wrong and I didn't catch it until the live
harness deadlocked** (see Step 6) - a `Task { await previous?.value }`
placeholder only waits for the *previous placeholder*, not the previous
call's actual engine work, so it resolves near-instantly regardless of
whether the real search finished, serializing nothing.
Fixed pattern: the queued unit of work is the real task itself
(`Task<EngineToolResult, Error> { await previousTail?.value; return try
await self.runCoachEvaluateSearch(...) }`), and `coachEvaluateTail` stores
a thin wrapper that awaits *that* task's completion - so the next caller
genuinely waits for the full previous call, not just its enqueue step.
App build + full `xcodebuild test` suite green with the (initially wrong)
fix in place, since nothing in the offline suite actually drove two
concurrent `coachEvaluate` calls - the live concurrent-evaluate probe
(step 6) is what caught it, which is exactly why the plan put that probe
in the live harness rather than only in unit tests.

### A real E2E-tooling lesson worth keeping

`scripts/axprobe.swift` (read) and `scripts/axclick.swift` (press, via
direct `AXUIElementPerformAction`) both work great without the app being
frontmost - useful for reading/clicking without stealing focus. Two
things do *not* work as hoped, discovered live this session:

1. **SwiftUI sheets don't render at all while the app is backgrounded**
   (launched via `open -j` / not activated) - confirmed by wiping the
   onboarding flag and polling `axprobe` against a non-activated process
   for several seconds with zero `AXSheet` elements ever appearing. Any
   sheet-driven flow (onboarding, Settings, the dashboard, chess.com
   fetch) needs the app frontmost just to exist in the AX tree, not only
   to be interacted with.
2. **`AXUIElementSetAttributeValue` on a text field's `AXValue`
   (`scripts/axsettext.swift`) does not drive SwiftUI's real `@State`
   binding** - it updates what `axprobe` reads back, convincingly, but
   the bound `@State` stays stale (confirmed: a "Check" button gated on
   `!username.isEmpty` stayed disabled after `axsettext` reported success
   and `axprobe` echoed the new value back). A real click - specifically
   a **CGEvent-posted click at real screen coordinates**
   (`scripts/axclickat.swift`), not a `System Events click` on the AX
   element and not a direct `AXUIElementPerformAction` press - was what
   actually handed the field real keyboard focus; `System Events
   keystroke` after that landed correctly and drove the binding for
   real. `System Events keystroke` on its own is also just a blind,
   frontmost-targeted global keyboard event regardless of which process
   you `tell` it to - a real hazard if the app isn't verified frontmost
   first (this happened live this session: an untargeted keystroke
   landed in the terminal chat instead of the app).

Net method going forward: reads and button/toggle/picker-menu presses
can stay frontmost-agnostic; any real text entry needs the app
genuinely activated, a real `CGEvent` click on the target field first,
then `System Events keystroke` - `axsettext.swift` is not a substitute
for that despite looking like one.
