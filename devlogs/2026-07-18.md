### Step 5 - Engine FIFO serialization

Added a `coachEvaluateTail` chokepoint to `EngineService.coachEvaluate`.
**First implementation was wrong and I didn't catch it until the live
harness deadlocked** (see Step 6) - a `Task { await previous?.value }`
placeholder only waits for the *previous placeholder*, not the previous
call's actual engine work, so it resolves near-instantly regardless of
whether the real search finished, serializing nothing.
Fixed pattern: the queued unit of work is the real task itself
(`Task<EngineToolResult, Error> { await previousTail?.value; return try
await self.runCoachEvaluateSearch(...) }`), and `coachEvaluateTail` stores
a thin wrapper that awaits *that* task's completion - so the next caller
genuinely waits for the full previous call, not just its enqueue step.
App build + full `xcodebuild test` suite green with the (initially wrong)
fix in place, since nothing in the offline suite actually drove two
concurrent `coachEvaluate` calls - the live concurrent-evaluate probe
(step 6) is what caught it, which is exactly why the plan put that probe
in the live harness rather than only in unit tests.

### Step 6 - `coach-grounding` chat section

Extended the executable: a legal-proposal turn ("what if I played
<a real rank-2 line's move>?"), an illegal-proposal turn (a SAN probed
live against the real position to guarantee illegality, not hardcoded),
an open question, and a concurrent-evaluate probe (`async let` two
`evaluate()` calls at once).
**Two real bugs found on the first live run, both fixed before the gate
passed:**
1. The concurrent-evaluate probe deadlocked the whole harness (600s
   watchdog fired, exit 1) - this is what surfaced the FIFO bug above.
   `GroundingEngine` (the standalone executor this executable drives,
   since it can't import the App target) got the same broken-then-fixed
   FIFO treatment as `EngineService`.
2. My own harness's independent re-verification step was re-checking
   `.precheck` replies (the canned "X isn't a legal move..." template)
   through `CoachVerifier` as if they were LLM prose - which flagged a
   false "leak" on the *token the template is declining*, e.g. `"Qh8"
   isn't a legal move..."` gets read by the verifier as citing `Qh8`, an
   illegal line, since the verifier only sees a stripped string, not that
   the source is a closed template.
   Fixed: only re-verify `.coach` text (fallback and precheck text are
   both closed templates, not model output, and have no business being
   run back through the verifier).
Both fixes are real code changes, not test-relaxations - the second run
exited 0 with the illegal turn showing `source=precheck` and zero LLM
requests, the concurrent probe returning two sane distinct evals, and the
verifier genuinely firing (2 violations on the legal-proposal turn with
`qwen3:0.6b`, matching the "zero violations would be suspicious" gate).

- Coach-disabled offline state confirmed: toggling `coachEnabled` off (via

- **Ollama unavailability: a real, non-blocking gap found, not a
  regression.** `kill -9`-ing Ollama didn't demonstrate the fallback path
  live in this environment - Ollama.app auto-respawns via both a
  `launchctl` LaunchAgent and its own menu-bar Electron process fast
  enough that the in-flight request just succeeded against the freshly
  restarted server. Freezing it instead (`kill -STOP`, simulating an
  unresponsive-but-not-torn-down connection rather than a dead one) hung
  the chat turn for 3+ minutes with no client-side timeout firing at all,
  until `kill -CONT` let it complete normally - `OllamaClient` has no
  request timeout configured, a pre-existing characteristic shared by
  M6's narration path, not introduced by M7, but only now exercised by a
  genuinely-unresponsive-not-dead scenario. The app never crashed in any
  of this - it's a hang risk, not a crash risk - but it means "Ollama
  becomes unresponsive mid-chat" degrades to an indefinitely-disabled
  input rather than a timely fallback. Not fixed this session (out of
  the M7 execution plan's scope); flagged for M8 or a dedicated fix.
  The mid-stream-death -> fallback code path itself *is* covered, just at
  the unit-test level (`CoachChatTests.midStreamClientDeathFallsBackNever
  Throws`) rather than demonstrated live this session.
- AX pass: input field, Send, starter chips, jump buttons, and Clear are
  all real native controls and clickable via AX-element reference
  (positional lookup was needed since several buttons/text fields report

  does produce a real `AXOutline`/`AXRow`/`AXCell` structure (unlike the

  So `List` structurally exposes *rows*, not text content, at least via
  this AX querying method - a real, if partial, improvement over
  `ScrollView`, but it doesn't fully resolve the M5 gap. Worth revisiting
  with a different AX attribute (e.g. `AXValue` on the row rather than
  `name`) in a future session rather than assumed solved.
- `coach-grounding` exits 0 with `qwen3:0.6b` (post-fix run); every

  CoachKit/ChessCore/AnalysisKit/EngineKit) and the full `xcodebuild test`

### What's real vs. what's a documented gap

Nothing unverified ever rendered in any live test this session - every
chat reply was correctly labeled `coach`/`fallback`/`precheck` and the
DB rows matched what appeared on screen. The gaps found (open-question
tool-calling with tiny models, prose-quality-not-safety with
non-frontier models, the `List` AX-text limitation, no client-side Ollama
request timeout) are all quality/UX/robustness gaps, not safety gaps -
consistent with the M6/M7 design's own stated guarantee boundary.
